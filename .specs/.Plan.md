# 개선 계획 (.Plan.md)

`.specs`와 실제 코드베이스를 비교 분석한 결과, 아래 항목들이 개선 후보로 도출되었습니다.

---

## 🔴 Priority 1 — 스펙에 정의되어 있지만 미구현

### [완료] 1-1. 조회수 중복 방지 (쿠키 기반)
- **스펙**: "쿠키에 `viewed_posts` 배열 저장, 동일 게시글은 24시간 내 재카운트 방지"
- **현재**: `ViewCounter` 컴포넌트가 페이지 진입 시 무조건 +1 호출
- **개선**: 쿠키 또는 `sessionStorage`에 이미 조회한 게시글 ID를 기록하여, 같은 사용자가 반복 새로고침해도 카운트가 올라가지 않도록 방지

### 1-2. `auth.schema.ts` 미존재
- **스펙 (architecture.md)**: `schemas/auth.schema.ts` 파일이 문서화됨
- **현재**: 해당 파일이 존재하지 않음 (로그인은 NextAuth가 전담)
- **판단**: NextAuth가 인증을 전담하므로 별도 스키마가 불필요. 스펙에서 제거 완료

### 1-3. `image.action.ts` 네이밍
- **스펙**: `post.action.ts`, `comment.action.ts` 등 `.action.ts` 접미사
- **현재**: `post.ts`, `comment.ts`, `like.ts`, `image.ts`
- **판단**: 현재 코드의 단순 네이밍이 더 간결하므로 스펙을 코드에 맞춰 수정 완료

---

## 🟡 Priority 2 — 코드 품질 및 안정성 개선

### [완료] 2-1. Python, SQL 언어 지원 누락
- **스펙 (editor.md)**: CodeBlock 지원 언어에 Python, SQL 포함
- **현재**: `CodeBlockComponent.tsx`의 `LANGUAGES` 배열에 Python, SQL이 없음
- **개선**: `LANGUAGES` 배열에 `{ value: 'python', label: 'Python' }`, `{ value: 'sql', label: 'SQL' }` 추가

### [완료] 2-2. GA 커스텀 이벤트 미구현
- **스펙 (features.md)**: "커스텀 이벤트: 게시글 조회, 좋아요, 댓글 작성"
- **현재**: GA는 페이지뷰만 자동 추적, 커스텀 이벤트 미전송
- **개선**: `gtag('event', ...)` 호출을 좋아요 토글, 댓글 작성 시점에 추가

### [완료] 2-3. `useIntersectionObserver` 훅 미구현
- **스펙 (architecture.md)**: `hooks/useIntersectionObserver.ts` 문서화
- **현재**: 해당 파일 미존재. TOC에서 IntersectionObserver를 직접 인라인 사용
- **판단**: TOC 내부 인라인으로 충분하므로 스펙에서 제거 완료. 추후 재사용 필요 시 분리

### [완료] 2-4. `PostList.tsx` 컴포넌트 미존재
- **스펙**: `components/post/PostList.tsx` 문서화
- **현재**: 게시글 목록 페이지(`posts/page.tsx`)에서 직접 렌더링
- **개선**: 목록 렌더링 로직을 `PostList.tsx`로 분리하면 재사용성 및 테스트 용이성 향상

## 🟢 Priority 3 — 향후 고려 사항

### 3-1. 에디터 확장 (TreeDiagram / StructureBlock)
- **스펙**: TreeDiagram, StructureBlock 확장 정의
- **현재**: 미구현
- **판단**: Mermaid의 flowchart/mindmap으로 대부분 커버 가능. 실제 수요 발생 시 구현 검토

### [완료] 3-2. `ui/` 범용 컴포넌트 디렉토리
- **스펙**: `components/ui/` 하위에 Modal, Button, Card, Sidebar 등 Compound 컴포넌트
- **현재**: `ui/` 디렉토리 미존재. 각 컴포넌트가 인라인 스타일링
- **개선**: 프로젝트 규모가 커지면 공통 Button, Modal 등을 `ui/`로 분리하여 일관성 확보

### 3-3. 다크모드
- **스펙**: "라이트 모드 only (다크모드 없음)"으로 명시
- **현재**: 일부 컴포넌트에 `dark:` 프리픽스 클래스가 산발적으로 존재
- **개선**: 다크모드를 지원할 계획이 없다면 `dark:` 클래스를 정리하여 CSS 번들 크기 절감

### 3-4. 포트폴리오 데이터 외부화
- **현재**: 포트폴리오 프로젝트/스킬 데이터가 페이지 파일에 하드코딩
- **개선**: 향후 프로젝트가 많아지면 Supabase 테이블이나 MDX 파일로 분리 가능
